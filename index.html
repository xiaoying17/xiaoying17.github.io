冒泡排序

第一轮：从arr[0]开始，两个元素依次相互比较，将最大值冒泡到最后一位;第二轮相互比较的元素从arr[0]至arr[len-1],这一轮的最大值最后到了arr[len-1]处，经过len-1轮排序完成。

快速排序

可以选取数组的中间元素为pivot，将数组中小于pivot的元素放在pivot左边的left数组中，大于pivot的元素放在pivot右边的right数组中，然后依次递归。

以上两种为交换排序。

简单选择排序

每一轮都从操作数组的第一个元素开始为基准，选择出最小的元素，将这个元素与基准元素交换。
[7, 5, 3, 11, 2, 1] –> [1, 5, 3, 11, 2, 7] –> [1, 2, 3, 11, 5, 7] –> [1, 2, 3, 7, 5, 11] –>[1, 2, 3, 5, 7, 11]

堆排序

1:构建大顶堆(从最后一个非叶节点开始调整)
2:将堆的根节点与最后一个叶节点互换，在数组中的表现为最大值到了数组最后一位
3:重新调整堆，不考虑已经归位的元素。
注意，调整堆的过程也是递归的过程。比如在调整根结点后，其子节点如果不满足大顶堆的规则，则也要调整。

注意heapSort中的循环。
以上两种为选择排序。

直接插入排序

打牌的思路，每拿到一张牌与前面已经排好序的牌从右至左比较，插入到合适的位置。
[7, 5, 3, 11, 2, 1] –> [7][5, 3, 11, 2, 1] –> [5, 7][3, 11, 2, 1] –> [3, 5, 7][11, 2, 1]

希尔排序

优先比较距离较远的元素。核心在于间隔序列的设定。并没有针对每一轮gap后比较的结果进行合并，而是原始数组上，在每一个常数gap上，插入排序间隔为gap的元素。插入排序从1开始，希尔排序从gap开始，gap+1时相当于另一个插入排序，放在同一个循环中互不影响，因为间隔是一定的。

域名发散

http静态资源采用多个子域名，充分利用现代浏览器的多线程并发下载能力
域名发散的原因:浏览器为了防止服务器崩溃，对于每个域名的链接数一般是有上限的，因此PC时代对静态资源优化时，通常将静态资源分布在几个不同域，保证资源分域名存储，以提供最大并行度，让客户端加载静态资源更为迅速。

Web浏览器会限制每个域名的连接active connections的个数，当下载的资源数量超过这个限制时，页面加载时间就会变慢因为超额的资源在队列中等待加载。开发人员会将内容分割到多个子域中。由于浏览器重置每个域的连接限制，每个额外的域允许额外数量的活动连接。这使用户可以从更高的吞吐量从同一个来源检索文件。

Web浏览器必须在每个附加域上执行DNS查找，并保持与每个域的连接，导致初始加载时间更慢

在手机端上，解析DNS会到1s+；影响首屏时间；而在PC端DNS通常几十ms，PC可以存储很多的域名地址。针对手机端，域名数一般不超过5个。

DNS工作过程

用户在浏览器输入url后，电脑会发送一个DNS请求到本地DNS服务器。本地服务器一般由网络接入服务器商提供。
1.查找浏览器缓存。浏览器会检查缓存中有无域名对应的IP地址。若无，下一步。
2.查找操作系统缓存。浏览器从hosts文件中查找是否有存储的DNS信息。
3.查找路由器缓存。路由器一般有自己的DNS缓存。
4.查找ISP DNS缓存。网络服务供应商，本地区的域名服务器，网络配置中的DNS服务器地址。
5.迭代查询
1.本地DNS服务器将请求转发到互联网的根域 . 。
2.根域将所要查询域名的顶级域的服务器ip地址返回给本地DNS。
3.本地DNS根据返回的ip地址，向顶级域发送请求，顶级域再将二级域的ip地址返回给本地DNS。
….最终得到你想要的ip地址。
查找到对应的ip地址后，通过ip地址查找到对应的服务器，浏览器将用户发起的http请求发送给服务器。

浏览器渲染

发表于 2018-01-17

浏览器的结构

1.用户接口 包括地址栏、前进后退键、书签等，即除了主窗口之外的部分。
2.浏览器引擎 查询和操作渲染引擎的接口
3.渲染引擎 用于显示请求的内容。如果请求内容是HTML，它负责解析HTML和CSS，并显示解析后的结果。
4.网络 用于网络调用，如http请求。
5.UI后端 用于绘制基本组件如组合选择框对话框等。
6.JS解释器 用于解释执行JS代码。
7.数据存储

渲染引擎

渲染引擎即在浏览器窗口中显示所请求的内容。Firefox使用Gecko引擎，Safari和Chrome使用Webkit引擎。
渲染引擎在取得内容之后，解析HTML构建DOM树，构建Render树，布局Render树，绘制Render树。
渲染对象和Dom元素相对应，但这种对应关系不是一对一的，不可见的Dom元素不会被插入渲染树，例如head元素。另外，display属性为none的元素也不会在渲染树中出现（visibility属性为hidden的元素将出现在渲染树中）。

How browsers work

从输入一个url到页面出现的过程[转]

  1. 用户输入网址（假设是个html页面，并且是第一次访问），浏览器向服务器发出请求，服务器返回html文件；
  2. 浏览器开始载入html代码，发现＜head＞标签内有一个＜link＞标签引用外部CSS文件；
  3. 浏览器又发出CSS文件的请求，服务器返回这个CSS文件；
  4. 浏览器继续载入html中＜body＞部分的代码，并且CSS文件已经拿到手了，可以开始渲染页面了；
  5. 浏览器在代码中发现一个＜img＞标签引用了一张图片，向服务器发出请求。此时浏览器不会等到图片下载完，而是继续渲染后面的代码；
  6. 服务器返回图片文件，由于图片占用了一定面积，影响了后面段落的排布，因此浏览器需要回过头来重新渲染这部分代码；
  7. 浏览器发现了一个包含一行Javascript代码的＜script＞标签，赶快运行它；
  8. Javascript脚本执行了这条语句，它命令浏览器隐藏掉代码中的某个＜div＞ （style.display=”none”）。突然少了这么一个元素，浏览器不得不重新渲染这部分代码；
  9. 终于等到了＜/html＞的到来，浏览器泪流满面……
  10. 等等，还没完，用户点了一下界面中的“换肤”按钮，Javascript让浏览器换了一下＜link＞标签的CSS路径；
  11. 浏览器召集了在座的各位＜div＞＜span＞＜ul＞＜li＞们，“大伙儿收拾收拾行李，咱得重新来过……”，浏览器向服务器请求了新的CSS文件，重新渲染页面。

HTTP头部信息

1XX:信息，请求收到，继续处理
2XX:成功 200
3XX:重定向 304not modified 缓冲的版本已更新并且客户端应刷新文档
4XX:客户端错误 404not found
5XX:服务器错误 502Bad Gateway 错误的网关 504Gateway Timeout 网关超时
